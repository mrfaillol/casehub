## 6. Proposta sobre `services/orchestra` (item 3 do briefing)

**Recomendação: (b) extrair para repo próprio `mrfaillol/orchestra`.**

### Argumentos para (b) extrair

1. **Acoplamento zero com o core.** Nenhum módulo do casehub fora de `services/orchestra/` importa o coordinator. O dashboard do vps-monitor consome via HTTP/PM2, não Python.
2. **Ciclo de vida diferente.** Orchestra coordena agentes IA cross-runtime (Claude/Gemini/Perplexity/Antigravity); CaseHub é um produto SaaS jurídico. Têm release cadences, deploy targets, e equipes (você + futuros) **distintos**. Misturar inflaciona o blast radius do casehub.
3. **Plano Basic pós-pivô (24/Abr).** O orçamento de complexidade do casehub está direcionado para `vs-usage-report` + UI-first Design + Basic SaaS — orchestra fica fora dessa missão até FASE 2+.
4. **Higiene do `state.json`.** Em repo próprio, faz sentido manter `state.json` runtime fora do git via gitignore; em casehub, o state polui diffs e tem 23 KB volátil sem dono claro.
5. **CMD (`app.example.com`) já é o feeder de UI sanitizada.** Orchestra como repo independente facilita o pipeline CMD → orchestra → ledger sem confundir camadas.

### Argumentos contra (a) manter

- Custo zero hoje. Mas o custo é cumulativo: cada commit no casehub tocando algo perto de `services/` paga imposto de "este código ainda existe?".

### Argumentos contra (c) arquivar/remover

- **Crítico:** `services/vps-monitor/config.py` declara um processo PM2 `"orchestra"` que o oncall observa. Remover sem migrar quebra o dashboard `/orchestrator`. Só faz sentido depois de migrar para repo próprio + redirecionar o vps-monitor.

### Sequência sugerida (NÃO executar agora — só quando o Victor autorizar)

1. Criar `mrfaillol/orchestra` privado.
2. `git subtree split --prefix=services/orchestra HEAD -b export-orchestra` no casehub e push para o novo repo (ou `git filter-repo --subdirectory-filter services/orchestra`).
3. Adicionar `git submodule` ou apenas referenciar via deploy do PM2 separado.
4. Atualizar `services/vps-monitor/config.py` se necessário.
5. `git rm -r services/orchestra/` no casehub. State files saem do histórico via filter-repo.
6. Convocar `/council` (red line: deploy topology) **antes** de mexer em PM2 prod.

**Outputs decisórios solicitados:** **uma palavra** — `extrair`, `manter`, `arquivar`. Não execute nada antes do meu OK por escrito.
