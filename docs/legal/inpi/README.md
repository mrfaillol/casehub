# CaseHub software registration prep

This directory stores the technical manifest and operator notes for a future
INPI software registration of CaseHub.

## Scope

- Issue: `mrfaillol/casehub#413`
- Prepared snapshot commit: `50e8d4ace31b614098354ad576b836bd8ca4ed89`
- Allowlisted CaseHub Basic source snapshot artifact generated outside git:
  `/tmp/casehub-inpi/casehub-basic-source-snapshot-50e8d4ace31b614098354ad576b836bd8ca4ed89.zip`
- Snapshot SHA-256:
  `46fde2ce52a661d3814e728bcabd51ab5c86e84dbf06118a7c74f76150f10d07`

The ZIP is not committed because the technical documentation/code snapshot is a
confidential evidence artifact. Keep redundant backups in a private storage
location controlled by the rights holder before filing.

The snapshot is intentionally generated from an allowlist of CaseHub Basic
application/source paths, rather than by archiving the full repository and
trying to exclude risky paths. The allowlist includes application entrypoints,
configuration, migrations, middleware, models, routes, static assets,
templates, products, packaging manifests, dependency manifests, and project
metadata needed to identify the software. It does not include repo-wide legacy
services, operational scripts, docs, generated evidence, test results, reports,
office/PDF artifacts, runtime exports, archived template notes, template backup
files, generated static reports, or customer contact maps/data unless a future
human/legal review explicitly expands the allowlist.

The ZIP is generated by `docs/legal/inpi/build-inpi-snapshot.py`, which writes
files in sorted order with fixed timestamps and `ZIP_STORED` compression to avoid
Git/compressor-version drift in the recorded digest.

This is a conservative identification snapshot for software-registration
evidence, not a runnable deployment package.

The SHA-256 above identifies the preserved ZIP artifact generated on this
machine with Apple Git 2.50.1. If the ZIP is regenerated later, recalculate the
hash and update this manifest before copying any value into e-Software.

Operational implications:

- Filing is electronic through e-Software.
- The GRU for software registration uses service code `730`.
- The applicant must generate a digital hash summary from the technical
  documentation or source snapshot.
- The hash algorithm and hash value are entered in the electronic form.
- The rights holder, not INPI, is responsible for preserving the confidential
  technical documentation with integrity.

## Remaining human/legal steps

- Confirm the legal rights holder and applicant data for the designated legal
  entity.
- Generate/sign the INPI Declaração de Veracidade with the correct ICP-Brasil
  certificate.
- Move the allowlisted source ZIP from `/tmp` to private redundant storage before
  any protocol.
- Deliver the artifact to the designated legal representative only through a
  secure channel, not an open chat.
